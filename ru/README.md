# Projective Development Standard (PDS)

**Версия:** 0.2


## Описание
PDS — spec-driven методология проектирования и разработки программного обеспечения, оптимизированная для делегирования задач AI-агентам. 

Фундаментальная парадигма: **«Спецификация как Первоисточник»**. Исходный код классифицируется как транзиентный (временный) артефакт — проекция намерений архитектора. Удаление исходного кода не ведет к потере данных о системе; проект детерминировано восстанавливается на основе спецификаций.


## Архитектурная топология

Проект разделен на две строго изолированные области, связанные принципом структурного изоморфизма (зеркалирования директорий).

1. **Зона Определения (`[root]/pds/`)**: Единственный источник истины. Хранилище декларативных требований, ограничений и архитектурных решений. Управляется человеком (Архитектором).
2. **Зона Проекции (`[root]/...`)**: Исполняемый код, конфигурации. Генерируется и управляется PDS-Агентом.

### Схема PDS-совместимого проекта

```text
[root]/
├── pds/                              <-- ЗОНА ОПРЕДЕЛЕНИЯ (Source of Truth)
│   ├── -manifest-core.md             <-- Манифест (аксиоматика стандарта)
│   ├── -meta-backend.md              <-- Мета-спецификация (тех. стек, линтеры)
│   ├── -spec-config.json             <-- Спецификация реализации (иммутабельный текст)
│   └── src/
│       └── auth/                     <-- Изоморфная структура (отражает корень)
│           ├── -spec-login.md        <-- Функциональная спецификация (бизнес-логика)
│           └── -spec-crypto.ts       <-- Спецификация реализации (иммутабельный код)
│
├── src/
│   └── auth/
│       ├── login.ts                  <-- Артефакт синтезированный на основе -spec-login.md
│       └── crypto.ts                 <-- Артефакт синтезированный на основе -spec-crypto.ts
│
├── config.json                       <-- Артефакт синтезированный на основе -spec-config.json
├── package.json                      <-- Неуправляемый файл
└── README.md
```

## Система артефактов (PDS Quadrivium)

Зона Определения оперирует исключительно четырьмя типами файлов с каскадной системой приоритетов:

1. **Манифесты (`-manifest-[name].md`)**: Находятся только в корне `/pds/`. Задают глобальные правила методологии. Высший приоритет.
2. **Мета-спецификации (`-meta-[name].md`)**: Задают контекстные ограничения (стек, паттерны). Поддерживают иерархическое наследование для вложенных директорий.
3. **Функциональные спецификации (`-spec-[name].md`)**: Описывают смысл и бизнес-логику целевого модуля на естественном языке.
4. **Спецификации реализации (`-spec-[name].[ext]`)**: Фрагменты жесткого кода (интерфейсы, сигнатуры, критические алгоритмы). Копируются в Проекцию дословно. Модификация PDS-агентом запрещена.

## Ключевые механизмы

* **Избирательное управление (Selective Management):** PDS-Агент модифицирует только те файлы в Проекции, которые имеют прямое отображение в Зоне Определения. Остальные файлы игнорируются.
* **Семантическое якорение (ADR):** Архитектурные решения фиксируются непосредственно в тексте спецификаций или кода с помощью тега `@ADR: <обоснование>`. PDS-агент не имеет права игнорировать или удалять логику, защищенную данным тегом.

## Жизненный цикл

1. **Синтез (PDS → Code):** Трансляция намерений из Зоны Определения в исходный код Зоны Проекции силами PDS-Агента.
2. **Реконсиляция (Code → PDS):** Обратная синхронизация. Анализ ручных изменений в коде и конвертация их в обновления спецификаций. Требует обязательной валидации Архитектором.
3. **Верификация:** Автоматический аудит на предмет расхождений (Drift) между Зоной Определения и Зоной Проекции.
