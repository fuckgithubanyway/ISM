# PDS Manifest: Workflows & Protocols

**Версия:** 0.2
**Язык:** Русский


## Каскад Приоритетов

При возникновении противоречий (Коллизий) истиной считается источник с высшим приоритетом:

1.  **Манифест** (Методологический фундамент).
2.  **Мета-Спецификация** (Локальный контекст с учетом вложенности).
3.  **Спецификация Реализации** (Физический скелет).
4.  **Функциональная Спецификация** (Бизнес-логика).
5.  **Намерение пользователя** (Текущий запрос).

*@ADR: Высший приоритет Манифеста гарантирует, что Агент не сможет нарушить методологические принципы PDS (например, изменить логику в обход спецификации) даже при прямом запросе пользователя.


## Режимы Работы

### Синтез (Synthesis): `PDS` -> `Code`
1.  Агент читает каскад (Manifest + Meta + Impl Spec + Func Spec).
2.  Агент генерирует код в Зоне Проекции.
3.  **Правило Слияния:** Код из *Спецификации Реализации* копируется дословно (Immutable). Пропуски заполняются логикой, сгенерированной на основе *Функциональной Спецификации*.
*   *Область действия:* Применяется исключительно к файлам, имеющим соответствующие записи в `[root]/pds/`.
*   *Атомарность:* Агент должен стремиться к минимально необходимому изменению кода Проекции для соответствия Спецификации.

### Реконсиляция (Reconciliation): `Code` -> `PDS`
Используется при "ручном" изменении кода (хотфикс, отладка).
1.  Агент детектирует дрейф (Drift) между Проекцией и Определением.
2.  Агент формирует предложение по обновлению Спецификаций.
3.  Разработчик обязан зафиксировать обоснование согласно протоколу ADR (см. `-manifest-adr.md`).
*   *Статус:* Критическая операция с высоким риском семантической деградации.
*   *Протокол:* После автоматического обновления Спецификаций разработчик обязан провести верификацию сохранности и актуальности тегов `@ADR:`.
*@ADR: Определение реконсиляции как рискованной операции сознательно усложняет процесс "обратной правки", стимулируя разработчика работать в парадигме Specification-First.

### Верификация (Verification)
Автоматическая проверка соответствия Проекции Определению. Любое отклонение, не зафиксированное в Спецификации (отсутствующее в Impl Spec и противоречащее Func Spec), считается ошибкой.
