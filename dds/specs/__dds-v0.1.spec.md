# Development by Declarative Specification (DDS)

**DDS** — методология разработки, в которой Артефакты синтезируются Интерпретатором на основе декларативных Спецификаций.


## Онтология

*   **Артефакт (Artifact):** Объектная составляющая проекта: директория, символьная ссылка, или файл. Артефактами являются также и файлы Спецификаций.

*   **Спецификация (Specification):** Текстовое описание требований к системе. Является приоритетным источником истины ("Примат Спецификации"). Существует четыре вида Спецификаций:
    *   **Манифест (`__[версия].spec.md`):** Базовая Спецификация, определяющая протокол DDS. В проекте присутствует ровно в одном экземпляре.
    *   **Мета-Спецификация (`_[имя].spec.md`):** Описывает глобальный контекст или сквозные аспекты (архитектура, стек, стандарты кодирования) определенной части системы. Область действия Мета-Спецификации распространяется на директорию её размещения и все вложенные поддиректории (согласно зеркальной структуре каталога Спецификаций относительно корневого каталога проекта).
    *   **Функциональная Спецификация (`[имя].spec.md`):** Описывает функциональные требования, алгоритмы или структуру конкретного аспекта, Артефакта или модуля разрабатываемого продукта. Одна Функциональная Спецификация может описывать группу связанных Артефактов (например, заголовочный файл `.h` и реализацию `.cpp`, или компонент `.ts` и его тест `.test.ts`, и т.п.). Расположение Функциональных Спецификаций в директории `dds/specs/` повторяет (насколько это возможно) структуру проекта `root/`. Например, Спецификация `root/dds/specs/src/libs/math.spec.md` может описывать два Артефакта: `root/src/libs/math.h` и `root/src/libs/math.c`, или целый пакет `root/src/libs/math/`, со всеми целевыми Артефактами, что будет зависеть от контекста, и содержимого самой Спецификации.
    *   **Спецификация Артефакта (`[имя].[расширение].spec.md`):** Описывает требования к конкретному Артефакту (файлу). Ключевая особенность таких Спецификаций - колокация. Такая Спецификация располагается не в каталоге `dds/specs/`, а в файловой структуре самого проекта, в той же директории где и должен находится описываемый файл-Артефакт. Например, Спецификация `root/src/math.js.spec.md` описывает ровно один конкретный файл-Артефакт `root/src/math.js`, или `root/src/pages/login.html.spec.md` -> `root/src/pages/login.html` и т.п.

*   **Проекция (View):** Логический срез (подмножество) файловой структуры проекта, формирующий изолированный контекст для текущей итерации. Границы Проекции задаются соответствующим **Файлом Проекции** (конфигурацией), содержащим список ("Белый список") glob-шаблонов (на подобие `.gitignore`). Шаблоны разрешаются относительно директории расположения Файла Проекции. В ситуации недостаточного контекста, Интерпретатор вправе предложить более полный Файл Проекции, покрывающий контекст.

*   **Намерение (Intent):** Актуальная директива на естественном языке, определяющая цель текущей итерации. Конкретизирует задачу Интерпретатора применительно к переданному Снимку.

*   **Снимок (Snapshot):** Единое линеаризованное текстовое представление состояния проекта в определенной Проекции. Агрегирует структуру, содержимое файлов и Намерение. Может содержаться в текстовом файле с расширением вида `[название].ddssnapshot`. Для явного разделения содержимого отдельных блоков в тексте Снимка применяются заголовки-разделители вида: `---- DDS [НАЗВАНИЕ БЛОКА] ----`. Для минимизации коллизий названия заголовков начинаются с префикса `DDS`. Предпочтительный порядок следования блоков таков:
        `---- DDS PROJECT STRUCTURE ----` - Схема файловой структуры проекта;
        `---- DDS CONTEXT INDEX ----` - Список файлов текущей Проекции;
        `---- DDS VERIFICATION REPORT ----` - Лог работы статического анализатора проекта;
        `---- DDS MANIFEST: [путь/к/файлу] ----` - Манифест;
        `---- DDS META: [путь/к/файлу] ----` - Мета-Спецификация;
        `---- DDS SPEC: [путь/к/файлу] ----` - Спецификация Артефакта;
        `---- DDS CONTENT OF: [путь/к/файлу] ----` - Содержимое Артефакта;
        `---- DDS INFO OF: [путь/к/файлу] ----` - Информация о бинарном файле;
        `---- DDS INTENT ----` - Намерение для текущей итерации;
        `---- DDS END OF SNAPSHOT ----` - Метка окончания Снимка.

*Примечание:* В целях оптимизации размеров Снимка, массивы однородных числовых данных в текстовых Артефактах, подвергаются автоматической редукции. Место редуцирования помечается маркером `DDS REDUCED` (например: `[1, 2, 3, .... DDS REDUCED .... 98, 99, 100]`). Интерпретатор обязан воспринимать такие блоки как "существующие, но скрытые данные". Рекомендацией для пользователя является вынесение литеральных массивов в отдельные, исключаемые в Проекции, файлы.

*   **Интерпретатор (Interpreter):** Сущность (программный агент или оператор), осуществляющая когнитивную обработку Снимка. Функция Интерпретатора — сформировать **Отчет**, содержащий ответ на Намерение.

*   **Отчет (Report):** Текстовый документ представляющий собой последовательность **Блоков Решения**, и являющийся результатом работы Интерпретатора.

*   **Блок Решения (Solution Block):** Логически и синтаксически обособленная часть Отчета, описывающая один конкретный аспект решения задачи.
    *   Является единицей атомарности для процедуры применения изменений (Пользователь принимает или отклоняет Блок целиком).
    *   **Структура Блока:**
        1.  **Заголовок:** Маркер начала блока с кратким названием темы. Имеет вид: `==== DDS BLOCK: <ТЕМА> ====`.
        2.  **Описание:** Текст на естественном языке, объясняющий причины изменений, логику решения или результаты анализа.
        3.  **Директивы (Опционально):** Набор интегрированных в текст Описания формальных команд для модификации проекта.

*   **Директива (Directive):** Формальная инструкция на изменение какого-либо элемента проекта, интегрированная в Блок Решения. Подробно описаны в разделе "Протокол Директив".


## Топология проекта

Структура dds-совместимого проекта строго регламентирована в части хранения метаданных:

*   **`[root]/`**: Корень проекта. Содержит все целевые Артефакты проекта. Так же содержит колоцированные Спецификации Артефактов (`*.*.spec.md`).
*   **`[root]/dds/`**: Служебная директория DDS.
*   **`[root]/dds/specs/`**: Единое хранилище всех Спецификаций проекта.
    *   Содержит Манифест `__*.spec.md`.
    *   Содержит Мета-Спецификации `_*.spec.md`.
    *   Содержит Функциональные Спецификации `*.spec.md`.
*   **`[root]/dds/tools/`**: Директория для хранения операционных инструментов.

*Примечание:* В отношении Мета и Функциональных Спецификаций допускается любая вложенность директорий для логической группировки.

*Примечание:* Файлы Проекций (`*.ddsview`) могут располагаться в любой директории проекта, удобной для разработчика.


## Иерархия приоритетов

Элементы DDS-совместимого проекта имеют следующую иерархию приоритетов:

0.  **Намерение:** (`DDS INTENT`) всегда имеет наивысший приоритет.
1.  **Манифест:** (`__*.spec.md`) имеет высший приоритет над всеми Спецификациями и Артефактами.
2.  **Мета-Спецификация:** (`_*.spec.md`) задающая общие, сквозные правила и требованя.
3.  **Функциональная Спецификация:** (`*.spec.md`) задающая требования, алгоритмы или структуру конкретного аспекта.
4.  **Спецификация Артефакта:** Колоцированная Спецификация конкретного Артефакта (`*.*.spec.md`) описывающая более конкретные требования к конкретному Артефакту.
5.  **Артефакт:** Конкретный Артефакт проекта.


## Каскад Требований (Cascading Requirements)

При формировании конечного набора требований к Артефакту применяется принцип **наследования с переопределением**. Правила применяются в следующем порядке (от наименее приоритетных к наиболее приоритетным):

1.  **Мета-Спецификация (`_*.spec.md`):** Базовый слой. Задает глобальные стандарты (отступы, линтеры, архитектурные паттерны).
2.  **Функциональная Спецификация (`*.spec.md`):** Групповой слой. Уточняет требования для конкретного модуля или компонента. Переопределяет конфликтующие пункты Мета-Спецификации.
3.  **Спецификация Артефакта (`*.*.spec.md`):** Атомарный слой. Описывает уникальные требования конкретного файла. Имеет право нарушать любые вышестоящие правила (например, для legacy-файлов или исключений из правил линтера).
4.  **Манифест (`__*.spec.md`):** Конституционный слой. Определяет сам протокол работы. Его требования (формат файлов, структура заголовков) не могут быть переопределены никакими спецификациями.
5.  **Намерение (`Intent`):** Оперативный слой. Прямая команда пользователя в текущей итерации имеет наивысший приоритет (может требовать временного нарушения спецификаций для экспериментов).






## Протокол взаимодействия

Процесс разработки является итеративным и состоит из следующих фаз:

1.  **Захват (Capture):** Пользователь определяет Намерение, выбирает релевантную Проекцию и формирует Снимок.
2.  **Синтез (Synthesis):** Интерпретатор анализирует Снимок и генерирует Отчет (содержащий Анализ и/или План действий).
3.  **Применение (Application):** Пользователь валидирует полученный результат и вносит изменения в проект.

*Примечание:* При отсутствии явно описанного Намерения, предполагается, что Интерпретатор произведет аудит предложенного Снимка.

*Примечание:* Интерпретатор не гарантирует атомарность транзакций файловой системы. Откат неудачных изменений (Rollback) и контроль версий (Git) целиком лежат в зоне ответственности пользователя.


## Протокол Директив

Директивы являются исполнимой частью Блока Решения. Каждая Директива обрамляется маркерами начала и конца.

### Общий синтаксис

Любая директива начинается с маркера `==== DDS [КОМАНДА]: [АРГУМЕНТЫ] ====` и, если подразумевает наличие тела (контента), завершается маркером `==== DDS END ====`.

### Директива `CREATE` (Создание / Перезапись)
Используется для создания новых файлов или полной замены содержимого существующих.
*   **Логика:** Если файл по указанному пути существует, он будет перезаписан полностью. Если директории в пути отсутствуют, они должны быть созданы.
*   **Синтаксис:**
    `==== DDS CREATE: [путь/к/файлу] ====`
    `[Полное содержимое файла]`
    `==== DDS END ====`

### Директива `DELETE` (Удаление)
Используется для удаления файлов или директорий.
*   **Синтаксис:**
    `==== DDS DELETE: [путь/к/файлу] ====`

### Директива `MOVE` (Перемещение / Переименование)
Используется для изменения пути или имени Артефакта.
*   **Синтаксис:**
    `==== DDS MOVE: [старый/путь] ====`
    `==== DDS TO: [новый/путь] ====`

### Директива `UPDATE` (Контекстная Модификация)
Используется для внесения точечных изменений в текстовые файлы. Использует механизм "Поиск и Замена" (Search & Replace).
*   **Логика:** Интерпретатор находит в целевом файле фрагмент, указанный в блоке `SEARCH`, и заменяет его на фрагмент из блока `REPLACE`.
*   **Требования к блоку SEARCH:** Должен содержать достаточное количество уникального контекста (строк кода), чтобы однозначно идентифицировать место изменения в файле.
*   **Синтаксис:**
    `==== DDS UPDATE: [путь/к/файлу] ====`
    `==== DDS SEARCH ====`
    `[Точный текст заменяемого фрагмента (контекст)]`
    `==== DDS REPLACE ====`
    `[Новый текст фрагмента]`
    `==== DDS END ====`
*   *Примечание:* В рамках одной директивы `UPDATE` допускается несколько пар `SEARCH` / `REPLACE` для внесения множественных правок в один файл.


## Операционные инструменты

Для автоматизации работы с контекстом используются специализированные скрипты:

*   **`dds-snapshot [--help] [--intent "<текст>"]|[--intent-file <файл>] [--view <файл>] [--no-reduce] [--output <файл>]`:**
    Формирует Снимок, предварительно выполняя dds-verify и включая его отчет в структуру Снимка (`DDS VERIFY`).

*   **`dds-list [--help] [--view <файл>]`:**
    Формирует и ранжирует список файлов для включения в Снимок на основе правил файла Проекции. Обеспечивает корректный порядок следования контекста (Manifest -> Meta -> Spec -> Content).

*   **`dds-tree [--help] [--plain] [путь]`:**
    Генерирует текстовое представление файловой структуры проекта.

*   **`dds-verify [--help] [директория]`:**
    Проверяет структуру проекта на соответствие правилам DDS. Используется для обязательной предварительной верификации перед созданием Снимка.
