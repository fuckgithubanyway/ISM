
Development by Declarative Specification (DDS) — Разработка на основе Декларативных Спецификаций — методология разработки, при которой Артефакты проекта генерируются Исполнителем из декларативных Спецификаций на естественном языке.

Настоящий Манифест является исходной Спецификацией, определяющей протокол DDS, а его положения имеют абсолютный приоритет.


## Глоссарий

*   **Спецификация:** Текстовый файл, являющийся источником намерения. Содержит описание некоторого аспекта разрабатываемого продукта.
    *   **Манифест (`__[имя].spec.*`):** Базовая Спецификация, определяющая DDS протокол.
    *   **Мета-Спецификация (`_[имя].spec.*`):** Спецификация, описывающая глобальные, сквозные свойства проекта, такие как: архитектура, используемые языки, парадигмы программирования, стиль оформления кода, дополнительные аспекты Плана Изменений, и прочие сквозные аспекты.
    *   **Спецификация Артефакта (`[имя].spec.*`):** Спецификация, определяющая намерение для соответствующего ей Артефакта. Может описывать как один файл-Артефакт, так и группу связанных файлов, что должно прямо сообщаться в начале текста Спецификации, в перечислении вида:
        ```
        --- 
        Генерируемые Артефакты:
          - src/pages/help.html
          - src/pages/help.css
          - src/pages/help.js
        ---
        ```
        , или по шаблонам по типу:
        ```
        ---
        Генерируемые Артефакты:
          - *.c
          - *.h
        ---
        ```
        .

Рекомендуемой конвенцией является использование Markdown-формата для описания, и расширения `.spec.md` для файлов Спецификаций.

*   **Артефакт:** Объектная составляющая проекта: директория, символьная ссылка, или файл. Артефактами являются так же и файлы Спецификаций.

*   **Исполнитель:** Автоматизированный агент, который анализирует Спецификации и Артефакты из Снимка *(а так же Запрос, при его наличии)* для синтеза Плана Изменений.

*   **Снимок:** Текст, полученный путем сборки содержимого ряда файлов проекта в единый отформатированый документ. Для явного разделения содержимого отдельных блоков в тексте Снимка применяются заголовки-разделители *(для Markdown могут иметь вид: `--- [DDS-CONTEXT НАЗВАНИЕ БЛОКА] ---`)*. Названия заголовков начинаются с префикса `DDS-CONTEXT`, для минимизации коллизий. При наличии соответствующего `.ddsprotect` шаблона, содержимое Артефакта озаглавливается с флагом `PROTECTED` *(для Markdown могут иметь вид: `--- [DDS-CONTEXT НАЗВАНИЕ БЛОКА]: PROTECTED: [путь_к_файлу] ---`)*.
    *   **Полный Снимок:** Полный Снимок проекта инициализирующий DDS контекст для Исполнителя, и имеющий определенный порядок составляющих его блоков:
        Текст Манифеста, с заголовком `DDS-CONTEXT MANIFEST: PROTECTED: [путь_к_файлу]`;
        Схема файловой структуры проекта, с заголовком `DDS-CONTEXT PROJECT STRUCTURE`;
        Список файлов попавших в текущий Снимок, с заголовком `DDS-CONTEXT SNAPSHOT LIST`;
        Лог работы статического анализатора проекта, с заголовком `DDS-CONTEXT LINTER REPORT`;
        Все Мета-Спецификации, с заголовками вида `DDS-CONTEXT META: [путь_к_файлу]`;
        Все Спецификации Артефактов, с заголовками вида `DDS-CONTEXT SPEC: [путь_к_файлу]`;
        Все Артефакты, с заголовками вида `DDS-CONTEXT CONTENT OF: [путь_к_файлу]`;
        Запрос Действия, с заголовком вида `DDS-CONTEXT REQUEST`.
    *   **Ограниченный Снимок:** Снимок включающий только некоторые, определенные текстовые файлы проекта, и блоки. Сохраняет относительный порядок следования блоков, определенный для Полного Снимка.
Снимок *(как Полный так и Ограниченный)* завершается меткой `DDS-CONTEXT END OF SNAPSHOT`.

*   **Запрос Действия (Запрос):** Опциональная оперативная директива передаваемая Исполнителю как часто Снимка. Конкретизирует цель текущей итерации и является основным источником намерения в рамках текущего цикла.

*   **План Изменений (План):** Генерируемый Исполнителем формальный документ состоящий из последовательности Директив, описывающий последовательностью действий, необходимую для приведения проекта в соответствие Спецификациям.
    *   **Директива:** Формальная команда описывающая атомарную операцию над файлами разрабатываемого проекта. Список и формат Директив описывается в разделе "Протокол Плана Изменений".

*   **Дефект Спецификации:** Обнаруженная Исполнителем аномалия в Снимке, которая препятствует однозначному и корректному синтезу. Классифицируется по трем основным типам:
    *   **Противоречие:** Наличие двух или более логически несовместимых утверждений.
    *   **Двусмысленность:** Формулировка, допускающая несколько взаимоисключающих трактовок.
    *   **Неполнота:** Отсутствие критически важной информации для выполнения синтеза.

*   **Операционные Инструменты (Скрипты):** Ряд скриптов, используемых для автоматизации DDS-специфичных операций, как формирование Снимка, применение Плана, и т.п.


## Структура проекта

Все Спецификации располагаются в директории `dds/specs/` в корневом каталоге проекта. Файловая структура директории `dds/specs/` зеркально отражает структуру Артефактов в корневой директории. Каждая Спецификация определяет один или несколько связанных Артефактов.

**Пример:**

*   **Спецификация Артефакта:**
    `dds/specs/src/services/user_service.ts.spec.md`
*   **...определяет один Артефакт:**
    `src/services/user_service.ts`

*   **Спецификация группы Артефактов:**
    `dds/specs/src/utils/math.spec.md`
*   **...в зависимости от содержания, определяет один или группу Артефактов (например):**
    `src/utils/math.h` и `src/utils/math.c`

*   **Спецификация группы Артефактов:**
    `dds/specs/src/pages/help.spec.md`
*   **...в зависимости от содержания, определяет один или группу Артефактов (например):**
    `src/pages/help.html`, `src/pages/help.css` и `src/pages/help.js`


### Обработка `.ddsignore`

Опциональные текстовые файлы `.ddsignore` определяют шаблоны имен файлов и директорий, исключаемых из Снимка. Синтаксис этих файлов является **подмножеством** формата `.gitignore` и обрабатывается следующим образом:

*   **Комментарии и пустые строки:** Строки, начинающиеся с `#`, а также пустые строки, полностью игнорируются.

*   **Контекст правил:** Правила действуют относительно директории, в которой находится файл `.ddsignore`.
    *   *Пример:* Правило `temp/` в файле `src/.ddsignore` проигнорирует путь `src/temp/`, но не `temp/` в корне проекта.

*   **Привязка к корню проекта:** Шаблоны, начинающиеся со слэша (`/`), действуют от корня проекта, независимо от расположения файла `.ddsignore`.
    *   *Пример:* `/logs` проигнорирует только директорию `logs` в корне, но не `src/logs`.

*   **Шаблон `*` (одна звездочка):** Соответствует любой последовательности символов, **кроме слэша (`/`)**. Используется для совпадений в рамках одного сегмента пути.
    *   *Пример:* `*.log` проигнорирует `debug.log` и `error.log`, но не `logs/app.log`.
    *   *Пример:* `src/*.js` проигнорирует `src/index.js`, но не `src/utils/helpers.js`.

*   **Шаблон `**` (две звездочки):** Соответствует любой последовательности символов, **включая слэши (`/`)**. Используется для рекурсивного поиска по директориям.
    *   *Пример:* `**/node_modules` проигнорирует все директории `node_modules` на любом уровне вложенности.

*   **Директории:** Шаблоны, оканчивающиеся на слэш (`/`), применяются только к директориям.
    *   *Пример:* `build/` проигнорирует директорию `build` и все ее содержимое, но не файл с именем `build`.

**Важное ограничение:** В отличие от полного синтаксиса `.gitignore`, **не поддерживаются** шаблоны отрицания (начинающиеся с `!`).


### Обработка `.ddsprotect`

Опциональные текстовые файлы `.ddsprotect`, определяют шаблоны имен файлов и директорий, защищенных от **любых** модификаций со стороны Исполнителя. Синтаксис идентичен синтаксису `.ddsignore`. Артефакты имеющие соответствующий шаблон в `.ddsprotect` помечаются флагом `PROTECTED:` в заголовке Снимка, и используются Исполнителем **только как контекст для чтения**.


## Цикл DDS

Цикл DDS — это формализованный процесс, состоящий из ряда последовательных фаз.

**Фаза Анализа**

Исполнитель анализирует предоставленный Снимок.

1.  **Построение Модели Намерения:** Исполнитель агрегирует все Спецификации из предоставленного Снимка и Запроса Действия *(при его наличии)* для формирования целостной модели *требуемого* состояния системы. На этом этапе выявляются Дефекты Спецификаций.
2.  **Построение Модели Состояния:** Исполнитель анализирует все существующие Артефакты из предоставленного Снимка для формирования модели *фактического* состояния.
3.  **Анализ Расхождений:** Исполнитель сопоставляет модель намерения с моделью состояния для определения необходимых изменений.

**Фаза Синтеза**

На основе результатов анализа Исполнитель формирует План Изменений. Содержимое каждого создаваемого или изменяемого Артефакта должно быть получено в результате диалектического синтеза:
1.  **Тезис:** Формулируется первоначальный вариант реализации, решающий основную задачу намерения.
2.  **Антитезис:** Предложенная реализация подвергается критическому анализу на предмет соответствия Мета-Спецификациям и другим релевантным требованиям. Выявляются слабые стороны, ошибки, потенциальные проблемы, и прочие недочеты.
3.  **Синтез:** Создается итоговый вариант реализации, снимающий выявленные на этапе Антитезиса противоречия.

**Важно:** Количество комментариев в генерируемом коде должна стремиться к минимуму. Идеальный код должен быть понятен без комментирования.

**Рекомендация для Исполнителя:** При манипуляциях с файлами проекта, Исполнитель должен стремиться сохранять согласованность файловых структур проекта и директории спецификаций. Так, к примеру, осуществляя перемещение/переименование *(директива `RENAME`)* Артефакта, важно переместить/переименовать соответствующую ему Спецификацию.

**Важно:** При наличии дефектов или ошибок Исполнитель не генерирует План, а выводит отчет об ошибке, по возможности сопровождаемый объяснением причины, и конструктивными предложениями по исправлению.


## Протокол Плана Изменений

План Изменений состоит из последовательности директив, определяющих манипуляции над Артефактами проекта. Директивы озаглавливаются заголовками вида `=== DDS-ACTION [ИМЯ_ДИРЕКТИВЫ][: ПАРАМЕТРЫ] ===`.

Любые изменения в проекте, включая изменения Спецификаций, должны быть выражены через явные директивы в Плане Изменений.

В процессе автоматического выполнения Плана формируется отчет о применении изменений, включая сообщения об ошибках.

**Блок `COMMENT` (опциональный)**: Содержит человекочитаемое комментарий. Этот блок игнорируется при автоматическом выполнении Плана, но полезен для разработчика.

```
=== DDS-ACTION COMMENT ===
[Произвольный текст с описанием изменений]
```
*   **Рекомендация для Исполнителя:** Каждый План Изменений должен начинаться с такого блока, содержащего краткое и емкое описание ключевых целей всего Плана.
*   **Важно:** Количество блоков-комментария, в результирующем Плане, должно быть минимально.

**Директива `MAKE DIR`**: Создает новую директорию (рекурсивно, если требуется).

*   **Формат:**
```
=== DDS-ACTION MAKE DIR: [путь_к_новой_директории] ===
```
*   **Поведение:** Создает директорию с использованием логики аналогичной `mkdir -p`. Операция завершается ошибкой, если по указанному пути уже существует файл (не директория) или если путь недопустим.
*   **Рекомендация для Исполнителя:** Использовать для подготовки структуры перед созданием файлов.

**Директива `UPSERT`**: Создает новый файл или полностью заменяет содержимое существующего.

*   **Формат:**
```
=== DDS-ACTION UPSERT: [путь_к_файлу] ===
[Полное содержимое файла]
```
*   **Поведение:** Если файл не существует, создает его; если существует, полностью перезаписывает.
*   **Рекомендация для Исполнителя:** Предпочтительная директива для генерации или полного обновления артефактов на основе синтеза из спецификаций.

**Директива `PATCH`**: Выполняет замену блока текста в существующем файле с использованием регулярных выражений или точного поиска.

*   **Формат:**
```
=== DDS-ACTION PATCH: [путь_к_существующему_файлу] ===
=== DDS-ACTION SEARCH: [паттерн] ===  # /регулярное_выражение/флаги или точный_блок_текста
=== DDS-ACTION REPLACE: [новый_текст] ===  # Может использовать $1, $2 и т.п. для ссылок на захватывающие группы
```
*   **Поведение:** 
    1. Если паттерн начинается и заканчивается на `/` (например, `/pattern/gim`), он трактуется как регулярное выражение с указанными флагами (поддерживаемые: `g` — глобальная замена всех вхождений, `i` — игнорирование регистра, `m` — многострочный режим, `s` — точка `.` совпадает с новой строкой).
    2. Иначе паттерн считается точным блоком текста (с учётом пробелов и переносов строк) для поиска и замены первого совпадения.
    3. Если поиск не найден или неоднозначен (множественные совпадения без флага `g`), операция завершается ошибкой.
    4. Для удаления блока секция `REPLACE` оставляется пустой.
    5. Поддержка ссылок на группы в замене (backreferences) для regex, что позволяет контекстно-зависимые изменения.
*   **Рекомендация для Исполнителя:** Включать достаточный контекст в поиск для уникальности. Использовать regex для гибкости, но всегда проверять на потенциальные ошибки (через тезис-антитезис-синтез). Уделять внимание пробельным символам и кодировке. Для глобальной замены обязательно указывать флаг `g`. Если требуется вставка без замены, использовать другие директивы, такие как `APPEND`.

**Директива `RENAME`**: Переименовывает или перемещает существующий файл или директорию.

*   **Формат:**
```
=== DDS-ACTION RENAME: [старый_путь] -> [новый_путь] ===
```
*   **Поведение:** Перемещает рекурсивно для директорий. Операция завершается ошибкой, если старый путь не существует или новый уже занят (без overwrite по умолчанию).
*   **Рекомендация для Исполнителя:** Сохранять файловую согласованность Спецификаций и Артефактов проекта.

**Директива `DELETE`**: Удаляет существующий файл или директорию.

*   **Формат:**
```
=== DDS-ACTION DELETE: [путь] ===
```
*   **Поведение:** Удаляет рекурсивно для директорий. Операция завершается ошибкой, если путь не существует.
*   **Рекомендация для Исполнителя:** Использовать для очистки устаревших артефактов после рефакторинга.

**Директива `LINK`**: Создает относительную символьную ссылку на исходный путь.

*   **Формат:**
```
=== DDS-ACTION LINK: [исходный_путь] -> [целевой_путь] ===
```

**Директива `COPY`**: Копирует существующий файл или директорию в новое место.

*   **Формат:**
```
=== DDS-ACTION COPY: [исходный_путь] -> [целевой_путь] ===
```
*   **Поведение:** Копирует рекурсивно для директорий. Операция завершается ошибкой, если исходный путь не существует или целевой уже занят.
*   **Рекомендация для Исполнителя:** Полезно для создания шаблонов или дублирования артефактов перед модификацией.

**Директива `APPEND`**: Добавляет текст в конец существующего файла.

*   **Формат:**
```
=== DDS-ACTION APPEND: [путь_к_существующему_файлу] ===
[Текст для добавления]
```
*   **Поведение:** Добавляет содержимое в конец файла. Операция завершается ошибкой, если файл не существует.
*   **Рекомендация для Исполнителя:** Использовать для инкрементальных обновлений, таких как добавление логов.

**Директива `SET ATTR`**: Устанавливает атрибуты файла или директории (например, права доступа).

*   **Формат:**
```
=== DDS-ACTION SET ATTR: [путь] ===
mode: [режим_доступа, например 0755]
```
*   **Поведение:** Применяет указанные атрибуты (рекурсивно для директорий). Операция завершается ошибкой, если путь не существует или атрибут недопустим.

**Директива `END`**: Сообщает о завершении Плана Изменений.

*   **Формат:**
```
=== DDS-ACTION END ===
```


## Операционные Инструменты

Для автоматизации Цикла DDS используются ряд основных скриптов:

*   **`dds-snapshot [--help] [--output <файл>] [--request <"текст"|файл>] [путь ...]` (Сборщик Снимка):**
    Формирует Снимок из указанных файлов и директорий, предварительно выполняя верификацию структуры проекта.

*   **`dds-apply [--help] [--input <файл>] [путь]` (Применитель Плана):**
    Интерпретирует и безопасно применяет предоставленный План Изменений к файловой системе проекта.

*   **`dds-run [--help] [--request <"текст"|файл>] [путь ...] -- <команда_LLM>` (Оркестратор):**
    Автоматизирует цикл формирования Снимка, передачи его Исполнителю, и применение сгенерированного Исполнителем Плана Изменений.

, а так же ряд вспомогательных:

*   **`dds-list [--help] [--all] [путь ...]` (Составитель Списка Файлов):**
    Формирует и приоритизирует список файлов для включения в Снимок, учитывая правила из `.ddsignore` и `.ddsprotect`. Обеспечивает корректный и предсказуемый порядок следования блоков в Снимке.

*   **`dds-tree [--help] [--all] [путь]` (Генератор Дерева Файлов):**
    Генерирует текстовое представление файловой структуры проекта.

*   **`dds-linter [--help] [директория]` (Линтер Структуры Проекта):**
    Проверяет структуру проекта на соответствие правилам DDS. Используется для обязательной предварительной верификации перед созданием Снимка.
