# Development by Declarative Specification (DDS)

**DDS** — методология разработки, в которой Артефакты синтезируются интеллектуальным Агентом на основе декларативных Спецификаций.


## Онтология

*   **Агент (Agent):** Когнитивная сущность (например, LLM или человек-оператор), осуществляющая анализ Снимка и синтез решений. Функция Агента — сформировать Отчет, удовлетворяющий Намерению.

*   **Артефакт (Artifact):** Объектная составляющая проекта: директория, символьная ссылка, или файл.

*   **Спецификация (Specification):** Текстовое описание требований к системе. Является приоритетным источником истины ("Примат Спецификации"). Все Спецификации DDS-совместимого проекта хранятся в каталоге `dds/specs`. Существует три вида Спецификаций:
    *   **Манифест (`__[версия].spec.md`):** Базовая Спецификация, определяющая протокол DDS. В проекте присутствует ровно в одном экземпляре.
    *   **Мета-Спецификация (`_[имя].spec.md`):** Описывает глобальный контекст или сквозные аспекты (архитектура, стек, стандарты кодирования) определенной части системы. Область действия Мета-Спецификации распространяется на директорию её размещения и все вложенные поддиректории (согласно зеркальной структуре каталога Спецификаций относительно корневого каталога проекта).
    *   **Функциональная Спецификация (`[имя].spec.md`):** Описывает функциональные требования, алгоритмы или структуру конкретного аспекта, Артефакта или модуля разрабатываемого продукта. Одна Функциональная Спецификация может описывать группу связанных Артефактов (например, заголовочный файл `.h` и реализацию `.cpp`, или компонент `.ts` и его тест `.test.ts`, и т.п.). Расположение Функциональных Спецификаций в директории `dds/specs/` повторяет (насколько это возможно) структуру проекта `root/`. Например, Спецификация `root/dds/specs/src/libs/math.spec.md` может описывать два Артефакта: `root/src/libs/math.h` и `root/src/libs/math.c`, или целый пакет `root/src/libs/math/`, со всеми целевыми Артефактами, или один Артефакт `root/src/libs/math.py`, что будет зависеть от контекста (Мета-Спецификации), и содержимого самой Спецификации.

*   **Проекция (View):** Логический срез (подмножество) файловой структуры проекта, формирующий изолированный контекст для текущей итерации. Границы Проекции задаются соответствующим **Файлом Проекции** (конфигурацией), содержащим список ("Белый список") glob-шаблонов (наподобие `.gitignore`). Шаблоны разрешаются относительно директории расположения Файла Проекции. В ситуации недостаточного контекста, Агент вправе предложить более полный Файл Проекции, покрывающий искомый контекст, и запрасить соответствующий Снимок.

*   **Намерение (Intent):** Актуальная директива на естественном языке, определяющая цель текущей итерации. Конкретизирует задачу Агента применительно к переданному Снимку.

*   **Снимок (Snapshot):** Единое линеаризованное текстовое представление состояния проекта в определенной Проекции. Агрегирует структуру, содержимое файлов и Намерение. Может содержаться в текстовом файле с расширением вида `[название].ddssnapshot`. Для явного разделения содержимого отдельных блоков в тексте Снимка применяются заголовки-разделители вида: `---- [ПРЕФИКС] [НАЗВАНИЕ БЛОКА][: путь/к/файлу] ----`. Для минимизации коллизий названия заголовков начинаются с префикса `DDS`. Предпочтительный порядок следования блоков таков:
        `PROJECT STRUCTURE` - Схема файловой структуры проекта;
        `CONTEXT INDEX` - Список файлов текущей Проекции;
        `VERIFICATION REPORT` - Лог работы статического анализатора проекта;
        `MANIFEST: [путь/к/файлу]` - Манифест;
        `META: [путь/к/файлу]` - Мета-Спецификация;
        `SPEC: [путь/к/файлу]` - Спецификация Артефакта;
        `CONTENT OF: [путь/к/файлу]` - Содержимое Артефакта;
        `INFO OF: [путь/к/файлу]` - Информация о бинарном файле;
        `INTENT` - Намерение для текущей итерации;
        `END OF SNAPSHOT` - Метка окончания Снимка.

*Примечание:* В целях оптимизации контекста, массивы однородных данных и длинные строковые литералы (Base64) подвергаются автоматической редукции. Место изъятия помечается маркером `.... DDS REDUCED [HASH/SIZE] ....`.

*   **Отчет (Report):** Текстовый документ представляющий собой последовательность **Блоков Решения**, и являющийся результатом работы Агента.

*   **Блок Решения (Solution Block):** Логически и синтаксически обособленная часть Отчета, описывающая один конкретный аспект решения задачи.
    *   Является единицей атомарности для процедуры применения изменений (Пользователь принимает или отклоняет Блок целиком).
    *   **Структура Блока:**
        1.  **Заголовок:** Маркер начала блока с кратким названием темы. Имеет вид: `==== DDS BLOCK: <ТЕМА> ====`.
        2.  **Описание:** Текст на естественном языке, объясняющий причины изменений, логику решения или результаты анализа.
        3.  **Директивы (Опционально):** Набор интегрированных в текст Описания формальных команд для модификации проекта.

*   **Директива (Directive):** Формальная инструкция на изменение какого-либо элемента проекта, интегрированная в Блок Решения. Подробно описаны в разделе "Протокол Директив".

*   **Инструментарий (Toolchain):** Набор служебных скриптов (`dds-*`), обеспечивающих детерминированные операции над файловой системой (чтение контекста, верификация, генерация Снимка).


## Топология проекта

Структура dds-совместимого проекта строго регламентирована в части хранения метаданных:

*   **`[root]/`**: Корень проекта. Содержит все целевые Артефакты проекта.
*   **`[root]/dds/`**: Служебная директория DDS.
*   **`[root]/dds/specs/`**: Единое хранилище всех Спецификаций проекта.
    *   Содержит Манифест `__*.spec.md`.
    *   Содержит Мета-Спецификации `_*.spec.md`.
    *   Содержит Функциональные Спецификации `*.spec.md`.
*   **`[root]/dds/tools/`**: Директория для хранения операционных инструментов.

*Примечание:* В отношении Мета и Функциональных Спецификаций допускается любая вложенность директорий для логической группировки.

*Примечание:* Файлы Проекций (`*.ddsview`) могут располагаться в любой директории проекта, удобной для разработчика.


## Каскад приоритетов

При формировании конечного набора действий применяется следующая иерархия (от высшего к низшему):

1.  **Манифест (`__*.spec.md`):** Конституционный слой (Immutable). Определяет формат данных, структуру заголовков и протокол обмена. Требования Манифеста **абсолютны** и не могут быть нарушены ни Намерением, ни Спецификациями.
    *   *Пример:* Агент не имеет права использовать нестандартные разделители блоков, даже если об этом попросит пользователь.

2.  **Намерение (`Intent`):** Оперативный слой (Command). Прямая директива пользователя для текущей итерации.
    *   Имеет приоритет над Функциональными и Мета-Спецификациями.
    *   Позволяет временно нарушать проектные правила (например, "Временно отключи линтер", "Используй хардкод для прототипа").

3.  **Функциональная Спецификация (`*.spec.md`):** Слой требований (Requirements). Описывает, **что** должно быть сделано.
    *   Переопределяет общие стандарты Мета-Спецификации для конкретного компонента.

4.  **Мета-Спецификация (`_*.spec.md`):** Контекстный слой (Environment). Описывает, **как** это делается по умолчанию (стиль кода, общие паттерны).



## Каскад приоритетов

При формировании Отчета и генерации кода Агент обязан руководствоваться строгой иерархией источников истины (от высшего к низшему).

1.  **Манифест (`__*.spec.md`) — Конституционный слой (Immutable).**
    *   Определяет формат данных, синтаксис протокола и структуру ответов.
    *   Требования Манифеста **абсолютны**. Они не могут быть нарушены или переопределены ни при каких обстоятельствах.
    *   *Пример:* Агент не вправе изменить формат Снимка, Отчета, или Директив.

2.  **Функциональная Спецификация (`*.spec.md`) — Законодательный слой (Specific Constraints).**
    *   Определяет конкретные требования к текущему модулю или Артефакту.
    *   Является **блокирующим ограничителем** для Намерения. Агент не имеет права генерировать код, противоречащий описанной функциональности.
    *   Переопределяет общие правила Мета-Спецификаций.
    *   *Разрешение конфликта:* Если Намерение требует действий, противоречащих Функциональной Спецификации, Агент обязан либо:
        а) Отказать в выполнении и указать на противоречие;
        б) Предложить предварительно обновить Спецификацию.

3.  **Мета-Спецификация (`_*.spec.md`) — Контекстный слой (General Constraints).**
    *   Определяет общие стандарты (стиль кода, архитектурные паттерны, технологии) для группы Артефактов.
    *   Действует как набор значений по умолчанию, обязательных к исполнению, если иное не указано в Функциональной Спецификации.

4.  **Намерение (`Intent`) — Исполнительный слой (Execution Trigger).**
    *   Инициирует процесс изменений. Определяет "Куда мы идем", но не "Какие законы нарушаем".
    *   Намерение должно реализовываться строго в границах, очерченных Спецификациями (п. 2 и 3).
    *   **Механизм `FORCE`:** Единственный способ нарушить приоритет Спецификаций (п. 2 и 3) — использование в Намерении маркера принуждения (*"FORCE: [текст намерения]"*). В этом случае Агент обязан выполнить директиву, добавив в Отчет предупреждение о нарушении целостности проекта.

5.  **Базовые модели — Справочный слой (Fallback).**
    *   Собственные знания Агента о языках программирования и best practices.
    *   Применяются только там, где поведение не определено в Спецификациях.





## Протокол взаимодействия

Процесс разработки является итеративным и состоит из следующих фаз:

1.  **Захват (Capture):** Пользователь определяет Намерение, выбирает релевантную Проекцию и формирует Снимок.
2.  **Синтез (Synthesis):** Агент анализирует Снимок и генерирует Отчет (содержащий Анализ и/или План действий).
3.  **Применение (Application):** Пользователь валидирует полученный результат и вносит изменения в проект.

*Примечание:* При отсутствии явно описанного Намерения, предполагается, что Агент произведет аудит предложенного Снимка.

*Примечание:* Агент не гарантирует атомарность транзакций файловой системы. Откат неудачных изменений (Rollback) и контроль версий (Git) целиком лежат в зоне ответственности пользователя.

*Примечание:* При выявлении критических проблем, противоречий или расхождений Агент должен сформировать отчет о ошибках содержащий так же и предложения по исправлению.


## Протокол целостности

Редуцированные блоки (`.... DDS REDUCED ....`) являются зонами повышенного риска потери данных. В их отношении действуют следующие законы:

1.  **Право на Структурный Вывод:** Агенту **разрешено** использовать видимые части файла (начало и конец массива) для определения типа данных и схемы.
    *   *Пример:* Видя `ids = [1, 2, .... DDS REDUCED ...., 99, 100]`, Агент вправе писать код, который итерируется по `ids` как по списку целых чисел.

2.  **Запрет Слепого Редактирования:**
    *   Строжайше запрещено использовать строку-маркер редукции (`.... DDS REDUCED ....`) внутри блока `SEARCH` директивы `UPDATE`.
    *   Запрещено пытаться "восстановить" или "дописать" данные внутри скрытой области.

3.  **Принцип "Read-Only" для Скрытого:**
    *   Если Намерение требует изменения *значений*, скрытых редукцией, Агент обязан остановить генерацию решения и запросить новый Снимок с флагом `--no-reduce`.
    *   *Исключение:* Допускается полное удаление файла (`DELETE`) или редактирование файла, если это не затрагивает редуцированные строки.


## Протокол Директив

Директивы являются исполнимой частью Блока Решения. Каждая Директива обрамляется маркерами начала и конца.

### Общий синтаксис

Любая директива начинается с маркера `==== DDS [КОМАНДА]: [АРГУМЕНТЫ] ====` и, если подразумевает наличие тела (контента), завершается маркером `==== DDS END ====`.

### Директива `CREATE` (Создание / Перезапись)
Используется для создания новых файлов или полной замены содержимого существующих.
*   **Логика:** Если файл по указанному пути существует, он будет перезаписан полностью. Если директории в пути отсутствуют, они должны быть созданы.
*   **Синтаксис:**
    `==== DDS CREATE: [путь/к/файлу] ====`
    `[Полное содержимое файла]`
    `==== DDS END ====`

### Директива `DELETE` (Удаление)
Используется для удаления файлов или директорий.
*   **Синтаксис:**
    `==== DDS DELETE: [путь/к/файлу] ====`

### Директива `MOVE` (Перемещение / Переименование)
Используется для изменения пути или имени Артефакта.
*   **Синтаксис:**
    `==== DDS MOVE: [старый/путь] ====`
    `==== DDS TO: [новый/путь] ====`

### Директива `UPDATE` (Контекстная модификация)
Используется для внесения точечных изменений в текстовые файлы. Использует механизм "Поиск и Замена" (Search & Replace).
*   **Логика:** Агент находит в целевом файле фрагмент, указанный в блоке `SEARCH`, и заменяет его на фрагмент из блока `REPLACE`.
*   **Требования к блоку SEARCH:** Должен содержать достаточное количество уникального контекста (строк кода), чтобы однозначно идентифицировать место изменения в файле.
*   **Синтаксис:**
    `==== DDS UPDATE: [путь/к/файлу] ====`
    `==== DDS SEARCH ====`
    `[Точный текст заменяемого фрагмента (контекст)]`
    `==== DDS REPLACE ====`
    `[Новый текст фрагмента]`
    `==== DDS END ====`
*   *Примечание:* В рамках одной директивы `UPDATE` допускается несколько пар `SEARCH` / `REPLACE` для внесения множественных правок в один файл.


## Операционные инструменты

Для автоматизации работы с контекстом используются специализированные скрипты:

*   **`dds-snapshot [--help] [--intent "<текст>"]|[--intent-file <файл>] [--view <файл>] [--no-reduce] [--output <файл>]`:**
    Формирует Снимок, предварительно выполняя dds-verify и включая его отчет в структуру Снимка (`DDS VERIFY`).

*   **`dds-list [--help] [--view <файл>]`:**
    Формирует и ранжирует список файлов для включения в Снимок на основе правил файла Проекции. Обеспечивает корректный порядок следования контекста (Manifest -> Meta -> Spec -> Content).

*   **`dds-tree [--help] [--plain] [путь]`:**
    Генерирует текстовое представление файловой структуры проекта.

*   **`dds-verify [--help] [директория]`:**
    Проверяет структуру проекта на соответствие правилам DDS. Используется для обязательной предварительной верификации перед созданием Снимка.
