# DDS Manifest: Structure & Topology

**Версия:** 0.5


## Корневая Директория (`dds/`)
Все артефакты методологии должны располагаться в выделенной директории `dds/` в корне проекта.
*   Эта директория содержит как конфигурацию методологии, так и дерево спецификаций.


## Типология Файлов

### Манифест-файлы (`-manifest-*.md`)
*   **Расположение:** Только корень `dds/`.
*   **Назначение:** Определяют фундаментальные правила методологии. Являются неизменяемыми для конкретного проекта (read-only reference), если не форкнуты намеренно.

### Мета-спецификации (`-meta-*.md`)
*   **Расположение:** Любая директория внутри `dds/`.
*   **Назначение:** Определяют контекст, правила и ограничения (Tech Stack, Code Style, Testing Rules, etc).
*   **Область действия:** Распространяются на директорию размещения и все вложенные поддиректории (рекурсивно). Мета-спецификации поддиректорий имеют приоритет над родительскими для всех Артефактов поддиректории (и для всех поддиректорий).

### Функциональные Спецификации (`[name].md`)
*   **Расположение:** Внутри дерева `dds/`, зеркально отражая структуру исходного кода.
*   **Назначение:** Описывают бизнес-логику, требования и контракты конкретного артефакта или модуля.

### Спецификации Реализации (`[name].[ext].md`)
*   **Расположение:** Внутри дерева `dds/`, зеркально отражая структуру исходного кода.
*   **Назначение:** Описывают технические детали реализации для конкретного языка или технологии.
*   *Пример:* `user.py.md` содержит специфику Python реализации конкретного Артефакта, в то время как `user.md` содержит общую логику.


## Принцип Зеркалирования (The Mirror Principle)
Структура директорий внутри `dds/` должна быть изоморфна структуре исходного кода проекта.

*   **Код:** `[root]/src/auth/login.ts`
*   **Спек:** `[root]/dds/src/auth/login.md`

Это обеспечивает алгоритмическую детерминированность поиска требований: Агент всегда знает, где искать Спецификацию для любого файла.
