# DDS Manifest: Context Resolution Protocol

**Версия:** 0.5


## Каскад Контекста (Context Cascading)
При работе над задачей Агент обязан сформировать **Контекст Выполнения**. Контекст собирается путем слияния файлов по иерархии от общего к частному.

### Алгоритм сбора:
Для целевого артефакта, расположенного по пути `src/A/B/file.ext`, Агент должен прочитать и применить правила в следующем порядке (от низшего приоритета к высшему):

1.  **Global Layer:** `dds/-manifest-*.md` (Фундаментальные правила).
2.  **Root Meta Layer:** `dds/-meta-*.md` (Глобальные настройки проекта).
3.  **Path Meta Layer:**
    *   `dds/src/-meta-*.md`
    *   `dds/src/A/-meta-*.md`
    *   `dds/src/A/B/-meta-*.md` (Наивысший приоритет среди мета-файлов).
4.  **Spec Layer:** `dds/src/A/B/file.md` (Бизнес-требования).
5.  **Implementation Layer:** `dds/src/A/B/file.ext.md` (Технические детали).


## Разрешение Конфликтов (Conflict Resolution)
*   **Затенение (Shadowing):** Правило (Мета-спецификация), определенное на более глубоком уровне вложенности, переопределяет (заменяет) аналогичное правило с верхнего уровня.
*   **Наследование (Inheritance):** Если правило не переопределено на нижнем уровне, применяется значение с верхнего уровня.
*   **Специфичность (Specificity):** `file.ext.md` имеет приоритет над `file.md`.
*   *Примечание:* Требования `file.ext.md` не должны противоречить требованиям `file.md`, а могут только уточнять детали конкретной реализации.


## Процедура Изменений (Modification Procedure)
Перед генерацией кода Агент обязан выполнить проверку:
1.  Есть ли изменения в Намерении (Intent), требующие правки логики?
2.  Если **ДА**:
    *   Шаг 1: Обновить соответствующую Спецификацию в `dds/`.
    *   Шаг 2: Сгенерировать или модифицировать затрагиваемые Артефакты, основываясь на новой Спецификации.
3.  Если **НЕТ** (рефакторинг, исправление ошибок без смены логики):
    *   Шаг 1: Убедиться, что текущее содержимое Артефакта соответствует Спецификации.
    *   Шаг 2: Внести необходимые изменения.

### Протокол Неявного Контекста (Implicit Context Protocol)

Определяет поведение Агента при работе с Артефактом, Спецификация для которого отсутствует.

#### Наследование Мета-контекста
Даже при отсутствии персональной Спецификации, Артефакт **подчиняется** всем Мета-спецификациям (`-meta-*.md`), действующим в его директории.
*   *Пример:* `file.ts` не имеет `file.md`, но обязан соблюдать Code Style из `dds/-meta-style.md` и использовать стек из `dds/-meta-stack.md`.

#### Инверсия Источника Истины (Fallback to Code)
Для Неспецифицированных Артефактов применяется принцип **"Code as Truth"**.
*   Агент считает текущую реализацию кода эталонным описанием его поведения (при отсутствии в содержимом явных ошибок, нарушений бизнес логики и иных недопустимых аспектов).
*   **Чтение:** При анализе проекта Агент извлекает логику непосредственно из исходного кода (Reverse Engineering).
*   **Модификация:** При запросе на изменение такого файла Агент обязан действовать в режиме **"Zero-Shot Modification"**:
    1.  Проанализировать текущий код.
    2.  Применить изменения согласно Намерению.
    3.  *Рекомендация:* Агент должен предложить пользователю создать базовую Спецификацию (`gen-spec`) для фиксации нового состояния.
