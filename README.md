
# DDS: Development by Declarative Specification
### Разработка на основе Декларативных Спецификаций

DDS — это методология разработки программного обеспечения, которая формализует и автоматизирует процесс генерации кода и других текстовых артефактов проекта с помощью интеллектуальных агентов, подобных LLM. Ключевая идея заключается в смещении фокуса разработчика с императивного кода («как делать») на декларативное описание требований к системе («что должно быть сделано»).

Цель подхода достигается за счет определения протокола взаимодействия между разработчиком и «Исполнителем» (LLM) через стандартизированные текстовые форматы:

*   **Намерение разработчика** выражается в виде набора текстовых **Спецификаций** на естественном языке.
*   **Контекст проекта** передается Исполнителю в виде **Снимка** — единого текстового представления релевантных файлов проекта.
*   **Результат работы Исполнителя** возвращается в виде машиночитаемого **Плана Изменений**, содержащего атомарные операции над файлами проекта.


## Как это работает: Цикл DDS

Рабочий процесс представляет собой простой и автоматизируемый трехэтапный цикл:

1.  **Описание.** Разработчик создает текстовые файлы Спецификаций (`*.spec.md`), описывающие те (некоторые или все) аспекты разрабатываемой системы, реализация которых делегируется **Исполнителю** (например, LLM).

2.  **Генерация.** С помощью **Операционных Скриптов** формируется **Снимок** проекта, который передается Исполнителю. Исполнитель анализирует запрос, и генерирует **План Изменений** — машиночитаемый список операций.

3.  **Применение.** Сгенерированный План передается на выполнение второму скрипту, который безопасно и интерактивно вносит изменения в файлы проекта.


## Структура DDS-совместимого проекта

DDS предлагает строгую и предсказуемую организацию файлов, где Спецификации отделены от Артефактов, но файловые структуры зеркально отражают друг друга.

```
some_dds_based_project/
│
├── .ddsignore   # ← Файл для исключения путей из Снимка
│
├── dds/
│   └── specs/   # ← Дериктория размещения Спецификаций
│       ├── __dds.spec.md   # ← DDS Манифест
│       ├── _project_architecture.spec.md
│       ├── src/
│       │   └── services/
│       │       └── user_service.ts.spec.md
│       └── configs/
│           └── database.json.spec.md
│
│   # ↓ Артефакты
├── src/
│   └── services/
│       ├── user_service.ts
│       └── other_service.ts
│
└── configs/
    └── database.json
```

В данном примере Артефакт `user_service.ts` является Управляемым, и генерируется Исполнителем, а Артефакт `other_service.ts` является Неуправляемым, не имеющим соответствующей Спецификации Артефакта, и находится под прямым управлением разработчика.


## Автоматизация и примеры использования

Ключевую роль в DDS играет автоматизация, обеспечиваемая Операционными Скриптами [`dds-snapshot`](./dds/scripts/dds-snapshot), [`dds-apply`](./dds/scripts/dds-apply) и  [`dds-run`](./dds/scripts/dds-run). Ниже представлены основные сценарии использования — от полностью ручного до максимально автоматизированного.

### Сценарий 1: Ручной режим (Copy/Paste) для веб-интерфейсов

Этот сценарий идеален, если у вас нет прямого CLI-доступа к LLM, но вы используете веб-интерфейс. Процесс состоит из двух шагов и не требует создания временных файлов.

*   **Шаг 1: Сгенерировать и скопировать Снимок в буфер обмена.**
    Выполните `dds-snapshot` и направьте его вывод напрямую в буфер обмена вашей системы.
    ```bash
    # Для Linux (требуется утилита xclip)
    dds-snapshot | xclip -selection clipboard
    ```
    Теперь откройте чат с LLM и просто вставьте содержимое буфера обмена.

*   **Шаг 2: Применить сгенерированный План из буфера обмена.**
    Скопируйте полный ответ LLM (План Изменений) в буфер обмена. Затем передайте его на вход скрипту `dds-apply`.
    ```bash
    # Для Linux (требуется утилита xclip)
    xclip -o | dds-apply
    ```
    Скрипт прочитает План напрямую из буфера обмена и безопасно применит изменения к вашему проекту.

### Сценарий 2: Полностью автоматизированный цикл с `dds-run` (Рекомендуемый способ)

Для полной автоматизации цикла `snapshot -> llm -> apply` предназначен скрипт-оркестратор `dds-run`. Он инкапсулирует всю логику, обеспечивает безопасность и предоставляет простой интерфейс.

**Синтаксис:** `dds-run [СОБСТВЕННЫЕ_ОПЦИИ_СКРИПТА] -- [КОМАНДА_ДЛЯ_LLM]`

Обязательный разделитель `--` сообщает скрипту, где заканчиваются аргументы для `dds-run` и начинается команда для вызова Исполнителя.

```bash
# Пример 1: Простой запуск с локальным скриптом-исполнителем
dds-run -- ./my-llm-script.sh

# Пример 2: Запуск с передачей Запроса и указанием конкретной директории
dds-run --request "Реализуй метод деления в калькуляторе" src/ -- ./my-llm-script.sh --model 'gpt-4o'
```


## О репозитории

*   **[`dds/specs/`](./dds/specs/)**: Коллекция готовых к использованию, версионированных **Манифестов** и **Мета-Спецификаций** (например, для описания стиля кода или архитектурных паттернов). Выберите подходящие и используйте их как отправную точку в своем проекте.
*   **[`dds/scripts/`](./dds/scripts/)**: Реализация **Операционных Скриптов**.


## Следующие шаги

Полная и формальная документация DDS подхода *(включая глоссарий, протокол Плана Изменений и другие аспекты)* представлена непосредственно в тексте канонического **[Манифеста](./dds/specs/__dds.spec.md)**. Так имеет смысл ознакомиться с интерфейсами ключевых Операционных **[Скриптов](./dds/scripts/)** `dds-snapshot --help`, `dds-apply --help` и `dds-run --help`.
